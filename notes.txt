1.Рендеринг страниц. 
Процесс называется Critical Rendering path (Критический путь рендинга).

Когда браузер получает от сервера файл с сухим текстом html, он превращает его в пиксели на экране за 5 шагов

1)Браузер начинает читать index.html сверху вниз для того что бы превратить текст кода в понятную для себя структуру объектов. 
С тегами напрямую браузер не работает. 
-Как только браузер видит тэг, он создает в памяти виртульный объект DOM (document object model/Объектная модель документа). 
Пример связи: 
Если в коде написано <body><div><p>Привет</p></div></body>, 
то в дереве DOM объект <body> станет «родителем», <div> — его «ребенком», а <p> — «внуком».

Строительство DOM-дерева происходит потоково. Браузеру не нужно ждать, пока файл докачается до конца.
Он скачал первые 10 строк — и тут же начинает строить из них верхушку дерева.

2)Парсинг Css и постороение CSSOM Дерева. 
Пока браузер читает HTML, он обязательно натыкается на тег <link rel="stylesheet">, который ведет к файлу со стилями (
например, style.css)
Скачивая это файл, браузер начинает строить новое дерево CSSOM(Css object model 
- Это дерево правил оформления. 
Браузер разбирает, какие стили к кому относятся. 
Причем правила наследуются (каскадируются): если для тега <body> задан шрифт Arial, то CSSOM автоматически пропишет 
этот шрифт и для всех блоков внутри него. 

3)Слияние в дерево рендинга (Rending tree). 
Теперь у браузера в памяти есть два независимых дерева: структура сайта (DOM) и правила красоты (CSSOM).
 На этом этапе он склеивает их в единое Render Tree (Дерево отображения).
Он идет по дереву DOM и проверяет для каждого элемента правила из CSSOM. 

Отсеивание невидимого: В Render Tree попадают только те элементы, которые пользователь реально увидит на экране.
Если у тебя в CSS для какого-то блока написано display: none;, то браузер полностью выкидывает этот блок из Дерева Рендеринга. 
Его как будто не существует.

4)Расчет геометрии (Layout/reflow)
(Компоновка, перекомпановка)
Браузер запускает мощный математический движок. Он берет ширину и высоту твоего окна Opera и начинает рассчитывать геометрию.
Этап 1. Layout (Компановка)
Браузер вычисляет где конкретно будет что находиться. 
Этап 2. Reflow / Перекомпоновка)
Процесс идет строго от родительских элементов к дочерним. Изменение размеров даже одной картинки может заставить браузер 
пересчитать геометрию всего сайта ниже. 

5)отрисовка (Painting) и Композиция (Composition). 


     2.Виды запросов и какие бывают. 

Когда ты вводишь адрес сайта в  браузере, твой компьютер не просто подлючается к серверу, 
Он отравиляет специальное сообщение HTTP-Запрос(Hyper Text Transfer Protocol).

Для того чтобы сервер понял чего именно хочет пользователь в запросах используются Специальные Методы(HTTP методы):
Методов Http запросов очень много, главные из них:

1)Get(получить)
Используется для запроса и чтения данных с сервера. 
Например клик по ссылке, шлешь get запрос. 
Как особенность, все параметры передаются прямо в адресной строке, после знака вопроса. 

2)Post(отправить)
Используется для отправки сложных или конфиденциальных данных на сервер (создание чего-то нового).
Данные упаковываются внутрь тела запроса (в посылку). Они скрыты от адресной строки, 
Именно через POST отправляются пароли при авторизации, файлы или длинные тексты писем (как в нашей форме с массивом $_POST). 

Так же Для примера:

PUT(Положить/заменить целиком). 

Этот метод используется для обновления данных на сервере, когда нужно полностью заменить существующий объект новыми данными
В базах данных он выполняет роль полной перезаписи строки через команду UPDATE. 

PATCH (Заплатка / Частичное изменение)
используется для частичного изменения объекта, когда нужно обновить только одно или два конкретных поля, не трогая остальные.\

DELETE (Удалить)Название говорит само за себя — этот метод используется для удаления ресурса с сервера 
 
В веб-разработке есть золотой стандарт CRUD 
(Create, Read, Update, Delete — Создать, Прочитать, Обновить, Удалить). 
Create (Создать)  метод POST  в SQL это INSERT 
Read (Прочитать) метод GET  в SQL это SELECT 
Update (Обновить) 
 методы PUT / PATCH  в SQL это UPDATE 
Delete (Удалить)  метод DELETE  в SQL это DELETE

1. Запросы.

Как передаются данные в POST. 

- в post запросах данные передаются внутри тела HTPP запроса(HTTP Body). Когда мы нажимаем кнопку отправки формы, 
браузер берет все данные, 
шифрует их в текстовую строку и прячет ее в скрытый от глаз конверт. Это происходит на 3 уровнях:
1)заголовок-указатель. Браузер обязательно добавляет в запрос служебную строку, которая сообщает серверу, 
в каком именно формате зашифрованы данные
2)Форматы кодирования

    - application/x-www-form-urlencoded (Стандартный): Все данные склеиваются в одну длинную строку, 
как в URL-строке (name=Alex&email=alex@mail.com), но прячутся в тело запроса

    -multipart/form-data: Используется, когда форма содержит файлы (например, пользователь загружает аватар или архив).
 Данные разбиваются на отдельные бинарные блоки. Если наставник спросит: «Как передать файл через POST?», 
правильный ответ — «Только сменив Content-Type на multipart/form-data».

    - application/json: Современный стандарт для API. Данные передаются в виде структурированного JSON-объекта {"name": "Alex"}.

3)Прием на бэкенде (PHP): Сервер Apache принимает этот поток байт, расшифровывает его в соответствии с Content-Type и 
автоматически наполняет суперглобальный массив $_POST (или массив $_FILES для картинок), с которым вы работаете в коде.

Главное отличие GET u POST. 
- GET идепонентный(безопасный). Он используется только для чтения, и сколько бы раз мы не обновляли страницу, на сервере ничегог не 
изменится. 
- POST не Идепонентный, он меняет состояние сервера (создает записи в БД, отправляет письма и тд.). 
- Ограничение по объему. GET запрос имеет лимит адресной строки (в зависимости от браузера, обычно 2048 символов).
У POST запроса лимита на размер тела со стороны протокола нет. Ограничения высставляются только со стороны php файла. 



2. Из чего состоит запрос, составные элементы. 

Любой HTTP запрос состоит из 3 элементов:

1)Стартовая строка/Start line/request line

Для запроса она всегда содержит три вещи: Метод (GET или POST), Путь/URL (/?controller=index) и Версию протокола (HTTP/1.1)
Пример: GET /?controller=index HTTP/1.1

2)Заголовки/HTTP Headers 
Это служебные метаданные в формате «Ключ: Значение». 
Они сообщают серверу информацию о браузере.
Примеры: Host: myshop.local (к какому сайту обращаемся), User-Agent: Opera/105.0... (какой у нас браузер), Accept-Language: ru-RU (какой язык мы предпочитаем).

3)Пустая строка (Обязательный разделитель)
Просто пустой перенос строки, который технически отделяет заголовки от контента.

4)Тело запроса(body). 
Тот самый «конверт». 
В GET-запросах тело всегда пустое. В POST-запросах здесь лежит текст из полей формы, JSON-данные или загружаемые файлы

3. Где подключаем CSS и JS файлы и почему?

1)CSS (Стили): Подключаем СТРОГО в теге <head>. 
Почему:
- CSS блокирует рендеринг страницы. 
Браузер должен построить дерево CSSOM до того, как начнет рисовать пиксели. 
Если подключить CSS внизу, пользователь на секунду увидит «голый», страшный и разломанный HTML-текст, 
который затем резко перекрасится (это называется эффектом FOUC).

2)JS (Скрипты): Подключаем в самом низу, перед закрывающим тегом </body>
Почему:
- Браузер, натыкаясь на <script>, намертво останавливает чтение HTML и ждет скачивания/выполнения JS-кода. 
Если подключить тяжелый JS вверху, сайт будет долго показывать белый экран. 
Подключение внизу позволяет пользователю сначала увидеть готовый HTML-дизайн, а уже потом в фоне загрузятся скрипты.

*Сейчас JS можно подключать и в <head>, но обязательно добавлять атрибут defer (например: <script defer src="script.js">). 
Этот атрибут приказывает браузеру скачивать скрипт в фоновом режиме, не блокируя чтение HTML, и запустить его только тогда, 
когда DOM-дерево будет полностью готово.*

4. Зачем нужна БД?

 БД нужна не просто для хранения, а для эффективного управления, безопасности и скорости.

1)Скорость и Индексы (Поиск):
Если в текстовом файле будет 10 миллионов пользователей, PHP будет читать его несколько минут. 
СУБД (MySQL/Postgres) благодаря индексам (B-Tree) найдет нужного юзера за микросекунды [mysql.com].
2)Параллельный доступ: 
Текстовый файл нельзя одновременно редактировать двум людям — он заблокируется. 
БД умеет одновременно обрабатывать тысячи запросов на чтение и запись от разных пользователей.
3)Связи и Целостность (Реляции): 
БД гарантирует, что у вас не появится заказ, привязанный к несуществующему пользователю. Файлы за этим следить не умеют.
4)Безопасность и Надежность (ACID): 
В СУБД встроен жесткий механизм транзакций. Если в момент перевода денег с одного счета на другой на сервере отключится электричество, 
БД либо выполнит операцию полностью, либо полностью отменит её, вернув всё назад. Данные никогда не зависнут в сломанном состоянии.

       ответы. 

1) ООП (объектно ориентированное планирование) 
Это способ борьбы со сложностью бльших проектов. оно позволяет объединить состояние (данные) и поведение (функции) в одну логическую единицу - класс. 
делая код модульным и масштабируемым. 

-класс (чертеж товара). мы пишем его один раз и закладываем в него структуру. у любого товара в нашем магазине должна быть цена, название и артикул. 
- объекты (физические товары в оперативной памяти). когда пользователь заходит в каталог, РНР делает запрос в базу данных и на основе нашего класса создает сотни независимых объектов. 

           4 принципа ООП:

1. Инкапсуляция (защита данных):
-мы прячем внутреннюю логику объекта от внешнего  мира, чтобы никто снаружи не мог внести изменение 
класс user имеет свойство $password
вместо $user->password = "123", мы пишем setPassword($newPassword), метод сначала захеширует пароль и проверит его свойства (длинну и тд). 

2. наследование (повторное использование). Возможность создать новый класс, на основе уже существующего. 
- есть класс user у него есть свойства $name, $email. Мы создаем класс Admin extends user, он автоматически получит все свойства user, 
но дополнительно мы пропишем ему функцию delete Pdoruct (). 

3. Полиморфизм (один интерфейс - много форм). Способность разных объектов выполнять одну и ту же команду. 
- есть метод pay(), если мы вызовем его для объекта Sber, то запрос полетит в сбербанк. для объекта yandex полетит к яндексу. и тд

4. Абстракция (выделение главного). 
- мы берем для объекта только те свойства реального объекта которые нам нужны. 
к примеру для переменной $user нам важны name,adress. цвет глаз размер обуви нас не интересует. 


          2)что такое интерфейс и используете ли вы его в своих проектах. 

Интерфейс - это строгий договор (или контракт) на то, что объект обязан уметь делать. Интерфейс не содержит в себе работающего кода или логики. 
Он содержит только список названий методов, которые класс обязан у себя создать.

*в коде своего текущего проекта я не писал их вручную, но я использую сторонние библиотеки (smarty), архитектуры которых полностью построены на интерфейсах.*

Если ты создаешь классы CdekDelivery или ПочтаРоссииDelivery, ты пишешь implements DeliveryInterface [php.net]. 
PHP жестко проследит: если ты забудешь написать в классе Почты функцию calculatePrice(), сайт выдаст критическую ошибку и не запустится [php.net]. 
Это гарантирует, что твой основной код сможет работать с любой доставкой одинаково.

API (Application Programming Interface — Внешний контракт)Это готовое окно или «дверь», 
которую один сайт или программа открывает наружу, чтобы с ней могли общаться другие сайты в интернете [w3schools.com].
Где находится: На удаленном сервере (например, сервер Сбербанка, Яндекса или ВКонтакте).Для кого нужен: Для связи между независимыми системами.
 Когда ваш интернет-магазин хочет принять оплату, ваш PHP-код отправляет HTTP-запрос (через методы POST или GET, которые мы разбирали) на специальный секретный
 адрес Сбербанка [w3schools.com]. Этот адрес и правила отправки запроса и называются API Сбербанка.Пример: Сбербанк открыл свое API. 
Ваш сайт отправляет туда JSON-данные с суммой заказа, API Сбербанка обрабатывает их на своем сервере и возвращает вашему сайту ответ: 
«Оплата прошла успешно» [w3schools.com]. При этом вы понятия не имеете, на каком языке написан сам Сбербанк (на Java, C# или Go) и как устроена их база данных.

        3)Что такое абстрактный класс и чем он отличается от интерфейса?

Абстрактный класс — это «полуфабрикат».  Это класс, который описывает общую сущность и общие правила для целой группы будущих объектов, 
но сам по себе существовать в памяти компьютера не может.Главное правило, которое нужно запомнить для наставника: 
*напрямую создать объект от абстрактного класса с помощью команды new нельзя (PHP выдаст критическую ошибку). 
От него можно только наследоваться.* В отличие от интерфейса, абстрактный класс может содержать в себе как готовые рабочие методы, 
так и пустые заготовки (абстрактные методы), которые обязаны дописать дети.

        4)Какие модификаторы видимости есть в PHP?

Модификаторы видимости (или управление доступом), как раз используются для реализации. они определяют из какого места программы можно прочитать или изменить переменную,
 запустить метод класса. Выделяют 3 основынх модификатора:

1)public - Доступен везде. внутри самого класса, в детях и в самом коде. 
2)protected (защищенный) - Доступен только «своим»: Переменную или метод можно вызвать только внутри самого этого класса И внутри классов-наследников (детей), 
которые от него отпочковались
3)private (приватный/закрытый) - Самая жесткая защита: Переменная или метод доступны строго внутри того конкретного класса, где они написаны. 
Даже классы-дети (наследники) не имеют к ним доступа.
